02

Convex spike

Treat Convex as a candidate source-of-truth platform, not just an iOS helper. The spike should prove whether Convex can replace most of the custom sync contract while GRDB still gives Lares the hard offline guarantee: edits survive app restart before the network returns.

Spike architecture

One vertical slice, intentionally small but realistic.

  • SwiftUI reads local GRDB projections.
  • GRDB stores editable notes, local patches, pending mutations, and status labels.
  • Convex Swift sends typed mutations when the device reconnects.
  • Convex subscriptions stream canonical notes and AI job state back to iOS.
  • Convex stores canonical note documents, patch receipts, files, and AI processing state for the spike.

Questions to answer

Each answer must come from working code, not just API reading.

  • Can an offline note edit survive app restart in GRDB and sync to Convex later?
  • Can Convex subscriptions replace our planned sync_changes feed?
  • Can Convex mutations cleanly apply, rebase, or reject note patches?
  • Can AI job state fit naturally in Convex without splitting ownership badly?
  • Does this reduce Lares Engine complexity instead of creating a second backend brain?

04

Fallback target shape

Keep this custom design as the fallback and comparison baseline. If Convex passes the spike, replace this section with a Convex-backed plan instead of building a duplicate sync engine.

iOS owns

The app surface owns fast local interaction and durable intent capture.

  • Editable local note documents, capture rows, and reactive query projections.
  • Drafts, attachment manifests, pending save state, and failed retry state.
  • sync_outbox rows for typed mutations owed to the Engine.
  • A SyncCoordinator actor that pushes commands and pulls facts.
  • No local note revision, extraction, proposal, or memory-writing behavior.

Engine owns

The Engine remains the durable runtime for product behavior.

  • Canonical journal and daily note document state.
  • Server-side mutation execution and deterministic patch merge/rebase.
  • Note revision creation and dedupe.
  • Agent run lifecycle, extraction, proposals, memory indexing, and summaries.
  • A schema-backed sync_changes feed for canonical read model updates.
  • A SyncChangeWriter seam behind persistence so workflows emit changes idempotently.

07

Sync contract

These endpoints must be schema-backed in apps/web/src/api-contract.ts and exposed through the generated OpenAPI/oRPC surface. Backend route tests and Swift decoder tests should share sample JSON fixtures.

POST /api/sync/mutations

Accepts user commands from the outbox.

  • note_document.patch applies local note edits against a base server revision.
  • capture.create records explicit captures and media events.
  • item.status.update changes a review item or action status.
  • proposal.review records accept, edit, reject, defer, or more-context decisions.
  • Same idempotencyKey plus same payload hash returns the prior result.
  • Same idempotencyKey plus different payload hash fails with idempotency_conflict.

GET /api/sync/changes

Returns canonical facts after a cursor.

  • Ordered by server-issued changeId, never device clocks.
  • Initial entities: journal document, capture event, agent run, extracted item, tombstone.
  • Next entities: proposal, item event, commitment, summary, synthesis.
  • Supports limit, hasMore, nextCursor, old cursor replay, and full backfill.
