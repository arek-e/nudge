# Observability From The First Slice

Lares must make request handling, Effect workflows, Workers Workflows, Cloudflare Agents activity, Human-in-the-Loop waits, and evaluation runs observable from the start. The first slice records request ids, elapsed timings, structured logs, Server-Timing headers, and Effect meters in `telemetry.ts` at the Hono edge; Effect services use `Effect.withSpan`, `Effect.annotateCurrentSpan`, `Effect.annotateLogs`, `Effect.log*`, and `Metric` meters; Workers Workflows use meaningful step names; Cloudflare Agents SDK diagnostics-channel events flow to Tail Workers in production; OTLP trace/log/metric export is added through Effect's observability layers once real Effect workflows need external export.

Logging follows the Wide Log principle: emit one structured completion event per meaningful unit of work with many queryable fields, including high-cardinality identifiers such as request ids, workflow ids, user-scoped ids, Review Queue item ids, Resume Token ids, and evaluation ids. Wide logs must not dump raw personal payloads, calendar descriptions, Relationship Memory content, model prompts, or unredacted Consent Grant contents; those remain source-linked data with explicit privacy controls.

At the Hono edge, middleware initializes a request-scoped wide event at request start, handlers enrich it through `addWideEventFields`, Hono `onError` enriches it with safe error fields, and the middleware emits exactly one canonical completion event at the end. The wide event also drives request metrics, so logs, metrics, and future trace spans describe the same unit of work rather than three disconnected stories.
