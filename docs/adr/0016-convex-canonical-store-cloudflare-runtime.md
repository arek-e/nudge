# Convex Canonical Store With Cloudflare Runtime

Nudge keeps Cloudflare Workers, Cloudflare Agents, and Workers Workflows as the runtime layer for HTTP/OpenAPI/MCP surfaces, live agent sessions, streaming, scheduling, tool execution, and retryable orchestration, but moves durable product data from D1 to Convex. Cloudflare Agent SQLite/state is runtime cache only: it may hold active stream status, pending connection metadata, and short-lived orchestration cursors, while user-visible or audit-relevant records such as messages, memories, proposals, reviews, tool runs, agent run outputs, workflow receipts, trace summaries, notes, and outcomes are stored in Convex.

This supersedes D1 as the canonical product store in the earlier Cloudflare-first architecture decisions. The tradeoff is an extra network/storage boundary from Workers, Agents, and Workflows into Convex, but the benefit is one realtime, typed, client-syncable product database across web, iOS, and Cloudflare runtime code without adopting the Convex Agent component or moving agent execution out of Cloudflare.

The migration uses a full Convex-backed adapter for the existing Effect `DbService` seam so routes, MCP tools, Agents, and Workflows can move without rewriting every caller at once. That adapter lives in `@nudge/db-convex`: `@nudge/db` remains the product storage port and legacy local adapters, while the Convex package owns the `ConvexRuntime` Effect context and `ConvexHttpClient` calls. The Worker owns generated Convex API references and provides `api.store` into that context. Worker runtime calls use `ConvexHttpClient` against `convex/store.ts` and include a `CONVEX_RUNTIME_SECRET`; Convex functions reject runtime store calls without the matching Convex environment secret. Browser/mobile realtime surfaces keep using auth-scoped Convex functions such as the daily note spike instead of this runtime adapter.

Every runtime Convex call carries W3C trace metadata in the Convex function args. Convex does not become the tracing runtime: it contributes deployment logs, function execution records, request IDs, and stack traces through the Convex Logs page or Log Streams. `convex/store.ts` logs a structured `convex_runtime_store_call` with the incoming trace ID, span ID, request ID, operation, route, service, and environment so Convex logs can be joined with Cloudflare logs and future external log streams. Cloudflare Agents and Workflows create or forward the same runtime trace context before running DB effects, so asynchronous analysis and agent tool calls stay connected to the originating HTTP trace when one exists.

D1 is no longer the durable product database and there is no runtime fallback from Convex to D1. Tests and narrow local seams may inject an explicit memory adapter, but deployed Workers, Agents, and Workflows fail fast unless `CONVEX_URL` and `CONVEX_RUNTIME_SECRET` are configured. Deployed Nudge runtime does not bind D1.
